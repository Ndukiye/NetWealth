# Deploying NetWealth

Split across two platforms, matching what each is good at:

- **API (`api/`)** → **Render** — a persistent Node service + managed
  Postgres. NetWealth's backend has a real cron job (`BankSyncScheduler`,
  every 5 minutes) which needs a long-running process — that doesn't work on
  Vercel's serverless functions, so the API doesn't go there.
- **Web (`web/`)** → **Vercel** — Next.js's own platform, zero config beyond
  pointing it at the `web/` subdirectory.

This is a monorepo with the API and web app as siblings, so both platforms
need to be told which subdirectory to build.

---

## 1. Push to GitHub

Both platforms deploy from a GitHub repo. If you haven't already:

```bash
git push -u origin main
```

---

## 2. API on Render

A `render.yaml` blueprint at the repo root already describes the API service
and its Postgres database, so this is mostly clicking through:

1. Render dashboard → **New** → **Blueprint**.
2. Connect the `Ndukiye/NetWealth` GitHub repo. Render reads `render.yaml`
   and shows two resources: `netwealth-api` (web service) and `netwealth-db`
   (Postgres).
3. A few env vars are marked `sync: false` in the blueprint (secrets Render
   won't auto-fill) — you can leave these **blank** for an initial deploy:
   `MONO_SECRET_KEY`, `OKRA_SECRET_KEY`, `OPENAI_API_KEY`,
   `TELEGRAM_BOT_TOKEN`, `CORS_ORIGIN`. They're only needed if you swap a
   mock integration for a real one (see `README.md` → "Swapping in real
   integrations"), or once you know the Vercel URL for `CORS_ORIGIN` (step 4
   below). `JWT_SECRET` and `BANK_WEBHOOK_SECRET` are auto-generated by
   Render — no action needed.
4. Click **Apply** / **Create New Resources**. Render builds
   (`npm install && npx prisma generate && npm run build`) then starts the
   service (`npx prisma migrate deploy && npm run start:prod`) — the
   `migrate deploy` step creates all the tables on the fresh database
   automatically.
5. Once it's live, note the URL Render assigns
   (e.g. `https://netwealth-api.onrender.com`) — you'll need it for the
   frontend's env var next.

**Seeding demo data (optional):** the blueprint only creates empty tables —
no demo user. To seed it, point your **local** machine at the Render
database temporarily and run the seed script from there (this works
regardless of Render plan, unlike using Render's Shell):

```bash
cd api
DATABASE_URL="<External Database URL from Render's netwealth-db page>" npx prisma db seed
```

Skip this if you'd rather sign up fresh through the app instead of using the
seeded demo account.

## 3. Web on Vercel

1. Vercel dashboard → **Add New** → **Project** → import the
   `Ndukiye/NetWealth` repo.
2. On the configure screen, set **Root Directory** to `web` (this is the
   monorepo step — without it Vercel tries to build the repo root). Framework
   preset should auto-detect as Next.js once you do.
3. Add an environment variable:
   - `NEXT_PUBLIC_API_URL` = `https://netwealth-api.onrender.com/api`
     (your Render URL from step 2.5, with `/api` on the end — that's the
     global prefix the NestJS app uses).
4. Deploy.
5. Copy the Vercel URL Vercel gives you (e.g. `https://netwealth.vercel.app`).

---

## 4. Lock down CORS

Now that you know the Vercel URL, go back to Render → `netwealth-api` →
**Environment** → set `CORS_ORIGIN` to your Vercel URL (no trailing slash;
comma-separate multiple origins if you have a preview URL too) → save. Render
redeploys automatically. Until this is set, the API accepts requests from
any origin, which is fine for a first deploy but worth tightening before
sharing the link around.

---

## Notes

- **Render free tier** web services spin down after ~15 minutes of
  inactivity — the first request after idle takes 30–60s to wake back up.
  Don't be alarmed if the first login attempt after a while looks stuck.
- **Render free Postgres** expires after 90 days on the free plan (Render
  will email you beforehand). Fine for a demo, not for anything long-lived.
- This is still the MVP scaffold described in `README.md`/`FEATURES.md` —
  auth hardening (refresh tokens, rate limiting) isn't done, so treat any
  deployed instance as a demo, not something to put real financial data
  into.
